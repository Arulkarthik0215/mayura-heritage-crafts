# Mayura Heritage Crafts - Order & Payment Flow

This document provides a simple, step-by-step breakdown of how the entire order and payment flow works in the system, from the moment the user clicks "Pay" to a successful checkout.

## 1. The Checkout Initiation (Frontend)
*   **User Action:** The customer fills in their contact information, shipping address, and clicks the **"Pay"** button on the `Checkout.tsx` page.
*   **Validation:** The frontend first checks if all required fields are filled out.
*   **Razorpay Setup:** It dynamically loads the Razorpay checkout script into the browser so the payment window can open later.

## 2. Secure Order Creation (Backend)
*   **API Call:** The frontend sends the customer's details and a list of the products in their cart (just the Product IDs and Quantities) to your backend (`POST /api/orders`).
*   **Price Verification:** Your backend is secure. It **does not trust** the frontend's prices. Instead, it looks up the actual price of each product directly from the database to prevent tampering.
*   **Cost Calculation:** It calculates the total amount, adding shipping charges based on the country (Free over ₹999 for India, ₹99 for smaller Indian orders, and ₹1499 for International). All money is converted to **paise** (multiply by 100) because Razorpay expects the smallest currency unit.
*   **Razorpay Link:** The backend talks to Razorpay's servers and creates an official "Razorpay Order".
*   **Database Save:** The backend saves this new order in your PostgreSQL database with a status of **"PENDING"**, keeping track of the `razorpayOrderId`. It sends this ID back to the frontend.

## 3. The Payment Modal (Frontend & Razorpay)
*   **Opening the Gateway:** The frontend receives the response from your backend and uses it to open the secure Razorpay payment window (`new window.Razorpay(...)`).
*   **The Transaction:** The user enters their card details, UPI, or Netbanking info and completes the payment on Razorpay's secure screen.

## 4. Payment Verification (Backend)
*This is the most critical security step to ensure the payment is authentic.*

*   **Success Callback:** Once the payment is successful, the Razorpay window gives the frontend a receipt containing three things: 
    *   `razorpay_order_id` (The ID of the order we created earlier)
    *   `razorpay_payment_id` (The unique ID for the actual money transaction)
    *   `razorpay_signature` (A complex string of characters generated by Razorpay's server)
*   **API Call:** The frontend immediately sends these three pieces of info to your backend (`POST /api/orders/verify`).
*   **Security Check (Crucial):** 
    *   **The Problem:** Without security, a malicious user or hacker could easily send a fake request to your backend saying "I paid for this order, here is my fake payment ID." If your backend just believed them, the hacker would get products for free without paying.
    *   **The Solution (Signature Verification):** To prevent this, Razorpay generates the `razorpay_signature` using a secret password (`RAZORPAY_KEY_SECRET`). Only Razorpay and your backend know this secret password; the customer never sees it.
    *   **How it works (The Mathematical Formula):** Your backend uses a cryptographic algorithm called **HMAC-SHA256** to generate its *own* signature. Here is the mathematical concept broken down:

        1.  **What is SHA-256?**
            SHA-256 (Secure Hash Algorithm) is a mathematical function that takes any input (like a string of text) and scrambles it into a unique, fixed-length 64-character string called a "hash".
            *   **It's One-Way:** You can turn "Hello" into a hash, but you can never mathematically reverse the hash back into "Hello".
            *   **It's Highly Sensitive:** If you change even one tiny letter or number in the input, the entire output hash changes completely.

        2.  **What is HMAC?**
            HMAC stands for **H**ash-based **M**essage **A**uthentication **C**ode.
            Normal SHA-256 just scrambles the text. HMAC scrambles the text **using a secret password** (in this case, your `RAZORPAY_KEY_SECRET`).

        **The Formula in Plain English:**
        *"Take the Order ID, put a line (`|`) next to it, and put the Payment ID after that. Now, put that combined text into a blender (SHA-256) along with our secret password (HMAC), blend it up, and spit out a 64-character text string (Hex digest)."*

        **In code (Node.js), it looks exactly like this:**
        ```javascript
        import crypto from 'crypto';

        // 1. Combine the Order ID and Payment ID with a pipe "|" symbol
        const body = razorpay_order_id + '|' + razorpay_payment_id;

        // 2. Generate the mathematical signature
        const expectedSignature = crypto
          .createHmac('sha256', process.env.RAZORPAY_KEY_SECRET) // Set the algorithm and secret key
          .update(body)                                          // Feed it the combined string
          .digest('hex');                                        // Output the result as a Hexadecimal string
        ```
    *   It then compares the `expectedSignature` it just generated with the `razorpay_signature` provided in the request.
    *   If they match perfectly, it proves with 100% certainty that the request actually came from Razorpay and that the payment is genuine. A hacker cannot forge this signature without knowing your secret key.
*   **Order Update:** Since the signatures match, the backend updates the order in your database. The status changes from **"PENDING"** to **"PAID"**, and it saves the exact time of payment.

## 5. Order Complete (Frontend)
*   **Confirmation:** The backend tells the frontend that the verification was successful.
*   **Cleanup:** The frontend clears the user's shopping cart.
*   **Redirection:** The user is redirected to the `/order-success` page, showing them their unique order number (e.g., `MHC-260428-A1B2C`).
